System Overview

Tulpo is a self-hosted communication platform. All data stays on your server. No telemetry, no third-party services, no cloud dependencies.

Single Binary
Bun runtime serves API, WebSocket, and static frontend from a single process. No nginx, no reverse proxy needed.
SQLite
Embedded database with WAL mode. No separate DB server. Data lives in a single file at /data/tulpo.db.
Real-Time
Native WebSocket with heartbeat, auto-reconnect, and multi-tab support via connection pooling.

Security Model

Honest assessment of what's implemented, what's missing, and what the risks are.

Password Security

Argon2id Hashing (Hardened)

State-of-the-art password hashing. timeCost: 5, memoryCost: 65536 (64 MiB). Memory-hard (GPU/ASIC-resistant), time-hard (brute-force resistant). Significantly above OWASP minimum recommendations.

Automatic Salting

Every password gets a unique, cryptographically random 16-byte salt. Generated automatically by Argon2id on each hash operation. The salt is embedded in the hash output string, so no separate salt column needed.

Server-Side Pepper

A secret pepper (TULPO_PEPPER env var) is appended to every password before hashing. Even if the database is stolen, hashes cannot be cracked without the pepper. Defense-in-depth layer.

Rate Limiting & Brute Force Protection

Auth routes rate-limited to 20 req/min per IP. Brute force protection with escalating lockouts: 5 fails → 15min, 10 fails → 1h, 20 fails → 24h. Tracked per IP and per email.

How Are Passwords Protected?

Three layers of protection:
  • Salt — fresh 16-byte random salt per hash. Two identical passwords produce different hashes. Rainbow tables useless.
  • Pepper — server-side secret appended before hashing. DB theft alone is insufficient to crack passwords.
  • Argon2id — 5 iterations, 64 MiB memory. Makes each hash attempt expensive for attackers.

Hash format: $argon2id$v=19$m=65536,t=5,p=1$<salt>$<hash>
Input: password + TULPO_PEPPER (pepper never stored in DB)

Known Risks & Mitigations

XSS could steal session tokens from localStorage

Risk: If an attacker injects JavaScript (XSS), they can read localStorage.getItem("tulpo_token") and hijack the session.

Mitigation: SvelteKit auto-escapes template output. No {@html} usage. Content-Security-Policy headers should be added in production.

Alternative: HttpOnly cookies would prevent JS access to the token, but add CSRF complexity in SPA mode.

Rate limiting is in-memory (resets on restart)

Risk: Rate limit and brute force counters are stored in-memory. Restarting the server resets all counters, allowing an attacker to bypass lockouts.

Mitigation: Argon2id with timeCost=5 provides natural rate limiting (~500ms per hash). For production, consider persisting counters to SQLite or Redis.

SQLite file-level security

Risk: Anyone with filesystem access to tulpo.db has full access to all data including password hashes.

Mitigation: Run in Docker with a named volume. Set file permissions (chmod 600). Argon2id hashes are extremely expensive to crack even if stolen. Pepper would add another layer.

WebSocket authentication

Risk: Token is passed as URL query parameter (/ws?token=...), which may appear in server logs.

Mitigation: Token is validated server-side. Connection is rejected if token is invalid or expired. The IDENTIFY flow provides a secondary authentication step after connection.

0HELLOServer → ClientSent on connect, contains heartbeat_interval

10DISPATCHServer → ClientEvent dispatch (READY, MESSAGE_CREATE, etc.)

Connection Lifecycle

// 1. Client connects
ws = new WebSocket("ws://host/ws?token=...")

// 2. Server sends HELLO
← { "op": 0, "d": { "heartbeat_interval": 30000 } }

// 3. Client sends IDENTIFY
→ { "op": 3, "d": { "token": "uuid-session-token" } }

// 4. Server validates, sends READY
← { "op": 10, "t": "READY", "d": { "user": { ... } } }

// 5. Heartbeat loop (every 30s)
→ { "op": 1, "d": null }
← { "op": 2, "d": null }

// 6. Server checks isAlive every 60s
//    Dead connections get closed with code 4002

Connection Pool

Structure: Map<userId, Set<WebSocket>>

Multi-tab: One user can have multiple WebSocket connections (one per browser tab). All receive the same events.

Cleanup: On disconnect or heartbeat timeout, the specific socket is removed. When a user has zero connections, they're removed from the pool.

Exponential backoff with 30-second cap. Resets to 1s after successful connection.

Database Schema

SQLite with WAL mode, foreign keys enforced, 5s busy timeout. All IDs are 24-char hex strings via lower(hex(randomblob(12))). Timestamps are ISO 8601 text.

SQLite Pragmas

PRAGMA journal_mode = WAL;      -- Write-ahead log for concurrent reads
PRAGMA synchronous = NORMAL;     -- Balance of safety and speed
PRAGMA foreign_keys = ON;        -- Enforce referential integrity
PRAGMA busy_timeout = 5000;      -- Wait 5s on lock contention

Deployment

Docker (Recommended)

Multi-stage build: dependencies → frontend build → production image.

# Stage 1: Install deps
FROM oven/bun:1.3 AS deps
COPY package.json bun.lock packages/*/package.json
RUN bun install --frozen-lockfile

# Stage 2: Build frontend
FROM deps AS frontend-build
COPY packages/shared/ packages/frontend/
RUN cd packages/frontend && bun run build

# Stage 3: Production
FROM oven/bun:1.3 AS production
COPY backend + shared + db + frontend/build
EXPOSE 3000
VOLUME /data
CMD ["bun", "packages/backend/src/index.ts"]
